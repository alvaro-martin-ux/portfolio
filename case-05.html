Case Study — RENFE UI System

A focused design system built for the Fare Selection screen — structured as four interdependent layers inside a single Figma file: primitive tokens, semantic aliases, a component library, and an assembled production screen.

Challenge
Design the Fare Selection screen without a pre-existing component library or token system
Approach
Primitive → Semantic → Component → UI token architecture, all inside a single structured Figma file
Deliverable
Figma file: 4 named layers, token documentation, component matrix, assembled production screen
Role
Product Designer — sole contributor
01 — Context

The Fare Selection screen is the most decision-dense step in the RENFE purchase funnel. It presents four tariff tiers simultaneously — each with a different price, a different flexibility policy, and different included services — and asks the user to commit. Getting the visual hierarchy wrong here directly increases drop-off.

Rather than designing the screen in isolation, the approach was to build it on top of a structured token architecture first. This meant establishing colour, typography, spacing, and border values as primitives, aliasing them into semantic roles, assembling reusable Figma components, and finally composing the production screen from those components. The file structure reflects this sequence exactly: four layers, each dependent on the one below it.

i

This is a personal design exercise, carried out independently and not subject to any NDA or confidentiality agreement. It uses publicly available RENFE product information solely as a realistic context. The purpose is to demonstrate how I conceptualise and structure design systems — the Figma file is a deliberately simplified exercise scoped to a single UI screen, showing the full token and component architecture applied to one focused product moment rather than an enterprise-wide system.

The complete design system — primitive tokens, semantic aliases, component library and assembled production screen — is available in the public Figma file.

02 — File Structure

The Figma file organises the system into four named layers that enforce a strict dependency chain. Components can only reference the layer directly below them — no skipping layers, no local overrides.

  • Primitive tokens Raw, unnamed values: full colour palette (Base, Primary/50–900, Neutral/50–900, Elevations), typography scales (font-size/100–400), spacing steps (space-1–6), border widths, and border radii. No semantic meaning — every hex value in the system lives here exactly once.
  • Semantic Tokens Purpose aliases that reference primitives by role: surface, text, border, shadow, icon (iconografía), typography, spacing, and strokes. Changing a colour theme means remapping semantics, not touching every component. This layer is what makes future theming possible without rework.
  • Component Library Figma components built exclusively from semantic tokens: Base/Button (Primary, Secondary, Tertiary × Default, Pressed, Disabled), Pattern/FareCard, Pattern/ActionRow, Layout/Topbar, Platform/BottomNavigation, Platform/SystemBar. No component references a primitive directly.
  • UI The assembled production screen: FareSelection. Composed entirely from Component Library items — no local styles, no hardcoded values. The screen is a composition, not a fresh design. Adding a new screen means assembling from the library, not starting over.
03 — Primitive Tokens

Primitives are the vocabulary. They define what values exist in the system, without assigning meaning to any of them. The colour system is built around two families — Primary (brand) and Neutral (structure) — each with six stops, plus Base and Elevation values.

Primary — rosa · magenta · wine

RENFE's brand palette runs from a pale blush at /50 to a deep wine at /900. The mid-tone /500 is the dominant action colour. The darker stops (/700, /900) provide pressed and high-contrast variants.

Semantic tokens translate raw values into named intent. A component consuming surface/brand doesn't need to know its value is primary/500. If the brand colour changes, the component inherits the update without modification.

The key discipline in semantic naming: names describe purpose, not appearance. surface/subtle communicates context; primary/100 communicates a visual value. The first survives a rebrand. The second doesn't.

05 — Component Library

The component library builds upward from semantic tokens. No component hardcodes a primitive value — all visual properties are consumed through the semantic layer. This guarantees that a future rebrand propagates from tokens through to every component automatically.

Base/Button — 3 variants × 3 states

Three hierarchy levels — Primary for the single dominant action, Secondary for supporting CTAs, Tertiary for text links. Nine total configurations, all documented in the Figma file with consistent naming.

Pattern/FareCard — the system's centrepiece

FareCard is the most domain-specific component and the one carrying the highest density of simultaneous decisions: tier name, price, feature list, selection state, and optional "Recomendada" badge. It must be scannable under high cognitive load and remain operable with one thumb on a phone.

Anatomy — 5 slots
  • Tier name → text/heading/card (16px Bold)
  • Price → text/display/price (21px Bold)
  • Feature list → text/body/primary + icon/brand check
  • Container border → border/subtle resting, border/selected active
  • Badge slot (optional) → surface/brand + text/on-brand
States & rules
  • Default → 1px border/subtle (neutral/300)
  • Selected → 2px border/selected (primary/500)
  • Badge → max one per list, "Elige Comfort" by default
  • State managed by parent screen — card is stateless
  • Price always right-aligned at display/price scale

Pattern/ActionRow

The ActionRow surfaces when a round-trip journey is detected, offering the option to choose different tariffs for each leg. It uses surface/subtle (primary/100) as background — distinct from the white screen surface without competing with the fare cards for attention. Its secondary CTA uses the Secondary button variant.

Layout & Platform components

  • Layout/Topbar Navigation bar: circular back button (icon + border/subtle stroke) + page title at text/heading/page scale. The back button reuses the icon-button variant of Base/Button — no new element invented for navigation.
  • Platform/BottomNavigation Three-slot OS navigation: back arrow, progress dot (primary/500 fill), grid icon. Platform components respect native conventions but consume the same semantic token set as product components.
  • Platform/SystemBar Status bar standardised across all screen variants: "9:41" time + signal + WiFi + battery. Defined once in the component library; every screen starts from the same baseline rather than recreating it.
06 — UI Screen

FareSelection is the output of the entire system. It is assembled from the Component Library — no local styles, no token values referenced directly from the screen layer. The screen is a composition; its visual consistency is a consequence of the system, not a manual effort.

07 — Key Decisions

Several decisions in the system require explanation because the artefact alone — the token or the component — doesn't show the reasoning that produced it.

TensionThe problem
  • Four simultaneous tariff options create choice overload — the cheapest and most expensive are €23,85 apart
  • The "Recomendada" badge carries commercial intent — it reads as manipulative if applied to more than one card
  • ActionRow competes visually with the fare list if styled too prominently
  • "Ver tabla comparativa" is a necessary escape hatch — but positioning it incorrectly pulls users away from the primary conversion path
  • Primary/500 is strong — using it for both the button and the selected card border risks visual noise
ResolutionToken-level answer
  • Price displayed at text/display/price (21px Bold) — immediately scannable at a glance, no need to read feature text first
  • "Recomendada" rule encoded in component documentation: maximum one badge per FareCard list, applied to the second tier by default
  • ActionRow uses surface/subtle (primary/100 — the lightest brand tint) — visible without competing with the white fare card surfaces
  • Compare link styled with text/brand at body scale — accessible, findable, but clearly subordinate to the Primary CTA
  • Selected border uses stroke/selected (2px) vs CTA button (solid fill) — stroke vs fill creates sufficient hierarchy between the two uses of primary/500
08 — Learnings

Building a system tightly scoped to one screen reveals constraints that a broad-scope system never surfaces. Every decision is immediately tested against a real context, real content, and real user choices.

01
Scoping to one screen forces precision, not simplicity.
A system with nothing to hide behind produces fewer tokens, better names, and more honest components. Every primitive and semantic token in this file exists because the FareSelection screen needs it. Nothing is speculative.
02
Semantic naming is the system's long-term investment.
surface/subtle survives a rebrand. primary/100 doesn't. Every token name written with purpose rather than appearance is a decision that doesn't need to be remade when the visual language evolves.
03
Documenting all states is where the system earns its value.
Defining Pressed and Disabled states for every button variant — not just Default — is the least glamorous part of the work. But it's the difference between a library that answers "how does this look?" and one that answers "how does this behave?". Without the full state matrix, the component is a sketch, not a specification.
04
The four-layer dependency chain makes problems traceable.
When a visual inconsistency appears on the assembled screen, the chain — UI → Component → Semantic → Primitive — gives a clear audit path. The problem is either a wrong token reference, a missing state, or a misnamed semantic alias. None of these require rebuilding from scratch. That traceability is what makes a system maintainable over time.
